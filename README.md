# InAppPurchases Plugin for NativePHP Mobile

Cross-platform in-app purchases for NativePHP Mobile:

- **Android** via Google Play Billing
- **iOS** via StoreKit 2

## Installation

```bash
composer require wilsonatb/in-app-purchases
php artisan vendor:publish --tag=nativephp-plugins-provider
php artisan native:plugin:register wilsonatb/in-app-purchases
php artisan native:plugin:list
```

## Requirements

- Android permission: `com.android.vending.BILLING`
- Android dependency: `com.android.billingclient:billing:8.3.0`
- iOS capability: In-App Purchase enabled in app target

## iOS Testing Setup

In-app purchases **will not work** without products registered in App Store Connect — even in the sandbox environment.

### 1. Register products in App Store Connect

1. Go to [App Store Connect](https://appstoreconnect.apple.com)
2. Navigate to your app → **In-App Purchases**
3. Create a product for each Product ID used in your app (e.g. `diwdesign.calculadoradiw.coins.1`, `diwdesign.calculadoradiw.coins.2`, `diwdesign.calculadoradiw.pro.1`)
4. For each product define:
   - **Type**: Consumable, Non-Consumable, Auto-Renewable Subscription, or Non-Renewing Subscription
   - **Reference Name**: any descriptive name
   - **Product ID**: must match exactly the string used in `queryProductDetails` / `buy*` calls

### 2. Sign paid applications agreement

In App Store Connect → **Agreements, Tax and Banking**:

- Accept the **Paid Applications Agreement**
- Complete banking and tax information

Without this, products will appear as `notFoundIds` even if they are correctly created.

### 3. Test with StoreKit in Xcode

For simulator / local development testing without App Store Connect sandbox accounts, use **StoreKit Testing in Xcode**:

1. Open the Xcode project generated by NativePHP
2. Go to **Product → Scheme → Edit Scheme → Run → Options**
3. Select **StoreKit Configuration** and add a `.storekit` configuration file
4. Define your products, prices, and test scenarios in the StoreKit configuration file

See [Setting up StoreKit Testing in Xcode](https://developer.apple.com/documentation/Xcode/setting-up-storekit-testing-in-xcode) for detailed instructions.

### 4. Test with Sandbox accounts (TestFlight / physical device)

For testing on physical devices or TestFlight builds without the Xcode StoreKit configuration:

1. Go to **App Store Connect → Users and Access → Sandbox Testers**
2. Create a sandbox tester (email + password)
3. On the iOS device: **Settings → App Store → Sandbox Account** and sign in with the sandbox tester credentials
4. Purchases will use the sandbox environment automatically

## Lifecycle (critical path)

Always run this flow in order:

1. `initialize`  
2. `isAvailable`  
3. `queryProductDetails`  
4. `buyConsumable` / `buyNonConsumable` / `buySubscription`  
5. handle `PurchaseUpdated`  
6. verify on your backend  
7. grant entitlement  
8. `completePurchase` if `pendingCompletePurchase === true`

> ⚠️ **Refund/revocation risk (test and production):** if completion is delayed or skipped, stores can automatically revoke/refund transactions. Complete immediately after successful verification + delivery.

## Usage

### PHP (Livewire/Blade)

```php
use Native\Mobile\Attributes\OnNative;
use Wilsonatb\InAppPurchases\Events\PurchaseUpdated;
use Wilsonatb\InAppPurchases\Facades\InAppPurchases;

InAppPurchases::initialize(autoSyncOnResume: true);

$available = InAppPurchases::isAvailable();
if (! ($available->available ?? false)) {
    return;
}

$products = InAppPurchases::queryProductDetails(
    ids: ['coins_100', 'pro_upgrade', 'pro_monthly']
);

InAppPurchases::buyConsumable([
    'productId' => 'coins_100',
    'id' => (string) str()->uuid(),
]);

#[OnNative(PurchaseUpdated::class)]
public function onPurchaseUpdated(array $purchases): void
{
    foreach ($purchases as $purchase) {
        if (! in_array($purchase['status'] ?? '', ['purchased', 'restored'], true)) {
            continue;
        }

        // 1) Verify with backend using token/id from payload
        // e.g. $purchase['verificationData']['server'] or $purchase['purchaseToken']

        // 2) Grant entitlement after verification succeeds

        if (($purchase['pendingCompletePurchase'] ?? false) === true) {
            $isConsumable = (bool) (
                $purchase['isConsumableHint']
                ?? (($purchase['completionHint'] ?? 'unknown') === 'consume')
            );

            InAppPurchases::completePurchase(
                purchaseId: (string) ($purchase['purchaseId'] ?? ''),
                isConsumable: $isConsumable,
                waitForResult: true,
                id: (string) str()->uuid(),
            );
        }
    }
}
```

### JavaScript (Vue/React/Inertia)

```javascript
import { InAppPurchases, Events } from '@wilsonatb/in-app-purchases';
import { on, off } from '@nativephp/native';

await InAppPurchases.initialize({ autoSyncOnResume: true });

const availability = await InAppPurchases.isAvailable();
if (!availability?.available) {
    // disable purchase UI
}

await InAppPurchases.queryProductDetails({
    products: [
        { productId: 'coins_100', productType: 'inapp' },
        { productId: 'pro_monthly', productType: 'subs' },
    ],
});

const purchaseHandler = async (payload) => {
    for (const purchase of payload.purchases ?? []) {
        if (!['purchased', 'restored'].includes(purchase.status)) continue;

        // 1) Verify with backend
        //    purchase.verificationData?.server or purchase.purchaseToken
        // 2) Grant entitlement

        if (purchase.pendingCompletePurchase) {
            const isConsumable =
                purchase.isConsumableHint ?? (purchase.completionHint === 'consume');

            await InAppPurchases.completePurchase({
                purchaseId: purchase.purchaseId,
                isConsumable,
                waitForResult: true,
                id: crypto.randomUUID(),
            });
        }
    }
};

on(Events.PurchaseUpdated, purchaseHandler);

await InAppPurchases.buySubscription({
    productId: 'pro_monthly',
    id: crypto.randomUUID(),
});

off(Events.PurchaseUpdated, purchaseHandler);
```

|---|---|
| `initialize(options?)` | Initialize native billing/store listeners |
| `isAvailable()` | Runtime store availability |
| `queryProductDetails(options)` | Query product metadata |
| `buyConsumable(options)` | Start consumable purchase |
| `buyNonConsumable(options)` | Start non-consumable purchase |
| `buySubscription(options)` | Start subscription purchase |
| `restorePurchases(options?)` | Trigger restore flow |
| `syncPurchases(options?)` | Refresh owned purchases |

| `getPendingPurchases()` | Pending completion snapshot |
| `getStatus()` | Plugin status and diagnostics |
| `countryCode()` | Storefront country code |

## Method parameter contracts

- `initialize({ autoSyncOnResume?: bool })`
- `isAvailable()`
- `queryProductDetails({ ids?: string[], products?: [{ productId: string, productType: "inapp"|"subs" }], id?: string })`
- `buyConsumable({ productId: string, offerToken?: string, accountId?: string, profileId?: string, applicationUserName?: string, id?: string })`
- `buyNonConsumable({ productId: string, offerToken?: string, accountId?: string, profileId?: string, applicationUserName?: string, id?: string })`
- `buySubscription({ productId: string, offerToken?: string, oldPurchaseToken?: string, replacementMode?: int, accountId?: string, profileId?: string, applicationUserName?: string, id?: string })`
  - `replacementMode` (Android only): `SubscriptionUpdateParams` replacement mode for subscription upgrade/downgrade (deprecated in Billing 8.x, kept as raw `int`). Values:
    - `0` = UNKNOWN_REPLACEMENT_MODE (default)
    - `1` = IMMEDIATE_WITH_TIME_PRORATION
    - `2` = CHARGE_PRORATED_PRICE
    - `3` = IMMEDIATE_WITHOUT_PRORATION
    - `4` = DEFERRED
    - `5` = CHARGE_FULL_PRICE
  - `oldPurchaseToken` (Android only): token from the existing subscription for upgrade/downgrade flows
- `completePurchase({ purchaseId: string, isConsumable?: bool, waitForResult?: bool, id?: string })`
- `consumePurchase({ purchaseToken: string })` (Android behavior; iOS no-op success for API parity)
- `restorePurchases({ id?: string })`
- `syncPurchases({ reason?: string })`
- `getPendingPurchases()`
- `getStatus()`
- `countryCode()`

## Completion mapping (hard rules)

Use event payload hints as truth:

- `pendingCompletePurchase` => completion required now
- `completionHint` => `consume` or `acknowledge`
- `isConsumableHint` => boolean for `completePurchase`

Product type mapping:

- **Consumable (`purchaseKind=consumable`, `productType=inapp`)**  
  call `completePurchase(..., isConsumable: true)` (Android consume)
- **Non-consumable (`purchaseKind=non_consumable`)**  
  call `completePurchase(..., isConsumable: false)` (Android acknowledge)
- **Subscription (`purchaseKind=subscription`, `productType=subs`)**  
  call `completePurchase(..., isConsumable: false)` (Android acknowledge)
- **iOS all types**  
  call `completePurchase` when pending; plugin performs StoreKit `finish()`

## `PurchaseUpdated` payload

Top-level event keys:

| Key | Meaning |
|---|---|
| `source` | Event origin (`purchase_result`, `purchases_updated_listener`, `unfinished`, `restore`, `complete_purchase`, `products_queried`, etc.) |
| `requestId` | Correlation id (can be empty) |
| `platform` | `android` or `ios` |
| `timestamp` | Epoch ms |
| `purchases` | Array of purchase payload items |
| `status` | Optional top-level status (e.g. `canceled`) |

Purchase item keys emitted by plugin:

| Key | Meaning |
|---|---|
| `status` | `pending`, `purchased`, `restored`, `consumed`, `error`, `unspecified` |
| `platform` | `android` or `ios` |
| `purchaseId` | Primary purchase reference |
| `purchaseToken` | Android purchase token or iOS transaction id |
| `productIds` | Product ids for this purchase |
| `productType` | `inapp`, `subs`, `unknown` |
| `purchaseKind` | `consumable`, `non_consumable`, `subscription`, `unknown` |
| `completionHint` | `consume`, `acknowledge`, `unknown` |
| `isConsumableHint` | boolean hint derived from `completionHint` |
| `pendingCompletePurchase` | whether app must call `completePurchase` |
| `transactionDateMs` | Epoch ms transaction date |
| `verificationData.source` | `google_play` or `app_store` |
| `verificationData.server` | backend verification token/id (Android: purchase token; iOS: transaction id) |
| `verificationData.local` | raw local payload/receipt (Android: original JSON; iOS: JWS representation for App Store Server API verification) |
| `platformData.android.orderId` | Android order id |
| `platformData.android.purchaseToken` | Android token |
| `platformData.android.purchaseStateCode` | Android raw state code |
| `platformData.ios.transactionId` | iOS transaction id |
| `platformData.ios.originalTransactionId` | iOS original transaction id |
| `purchaseState` | Android normalized purchase state |
| `acknowledged` | Android acknowledged flag |
| `autoRenewing` | Android subscription auto-renewing flag |
| `error` | Present on error items (`code`, `message`) |

## Notes

1. Verify on backend before granting entitlement.
2. Grant entitlement before completing purchase.
3. Complete purchase immediately when `pendingCompletePurchase` is true.
4. `pendingCompletePurchase` + `completionHint` are authoritative.
